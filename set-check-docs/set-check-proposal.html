Set Check

An in-house engine that reads our drawing set and checks what contractors buy, submit, and count against what we specified — before it's ordered or built. Windows shown as the worked example.

Prepared by Ray Date July 21, 2026 Decision Build as a BMS tab · start with windows
01

We're upstream — our set is everyone's source of truth

Our drawings are the document every project buys and builds from, and everyone downstream re-reads them by hand: a contractor re-counts our set to bid it, a developer buys products off our schedules, a supplier ships against our notes. Every time what gets bought or counted drifts from what we drew, it surfaces late — a failed inspection, the wrong product, a re-order — on a set carrying our stamp.

The same document that caused the drift can catch it. Set Check reads our drawings and checks what comes back against what we specified — two kinds of check: does the product match our spec, and does the quantity match our set. Windows are the clearest first case, so we'll walk the whole mechanism through them.

02

How it works — windows as the worked example

We spec window sizes in the schedule; the required U-factor comes out of our REScheck once we model the envelope. So on windows, the two things worth checking are:

Check 1

Size

Does the purchased unit match the opening size we drew in the window schedule?

Source: our window schedule
Check 2

U-factor

Does the purchased unit meet the U-factor our REScheck is based on — the code non-negotiable?

Source: our REScheck
The rule that keeps liability low The engine only ever checks what we actually specified. On windows that means we skip series · model · grille · color · operation — the developer's choice, not ours. The same discipline holds on any item: check our spec, ignore their preferences.

04

Why our liability stays low

  • It checks against our own documents — it never independently rules on New Jersey code.
  • Every result is flagged for a person to confirm; nothing is signed automatically.
  • Purchase and installation responsibility stays with the contractor who orders and installs.
  • It speeds up review we're already responsible for as architect — it doesn't add a new duty.
05

What it costs to run

$0/mo new

A new tab in the existing BMS, new tables in the database we already pay for, and our existing AI key. The only variable cost is a few cents of AI processing per check — which the fee covers many times over.

A separate standalone app would have added ~$35/mo in new subscriptions and a second product to maintain. We ruled that out — this rides on what we already run.

06

What it earns — it scales with what we build

Price tracks scope. Build only the easy compliance checks and it's a modest add-on; build up to real takeoffs — telling a developer how much trim and siding to buy — and each project is worth several times more. Every tier is pure-margin revenue on jobs we already have; each one just raises the ceiling.

08

The pilot — zero live-job risk

  1. Pick a completed DaSilva, Rodriguez, or Costello job — one where we already know how the windows turned out.
  2. Feed it the real window schedule, the real REScheck, and the vendor brochure that was actually used.
  3. Run the two checks and compare the output against what we know happened on that job.
Success looks like It confirms a compliant unit, and it flags a deliberately mismatched one — a wrong size, and a U-factor above the REScheck basis. Real firm data, nothing live on the line.
The decision we're asking for

Approve building Set Check as a BMS tab, starting with the windows pilot on one completed job. No new spend — the only investment is Ray's build and maintenance time.
